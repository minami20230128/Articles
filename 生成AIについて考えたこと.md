# 生成AIについて考えたこと
## 結論
- 今後伸ばしていくべきなのは要件定義・設計・コード読解力・品質保証・セキュリティなど  
- エージェントコーディングに関して身につけるべきなのは「正しくコーディングさせるルール作り」の仕方
- 自分が把握できない変更をさせ、テストも全部書かせてろくに確認もせず「どうせAIは正しいから」といってマージする人が見受けられるが、これはNG。
- **書き散らかして満足する人、思考停止してる人が一番要らない**

# 生成AIの出力には冪等性がない
冪等性とは、同じ処理を何度実行しても同じ結果が返ってくること。  
通常のソフトウェアでは、グローバル変数を使わないようにすれば冪等性を担保することは可能。  

OK例
```
int add(int num1, int num2){
    return num1 + num2;
}
```
NG例
```
static int NUM = 3;
int add(int num){
    return NUM + num;
}
```

## LLMの冪等性
LLMは出力が確率で変動するので、回答に冪等性はない。  
あるときは正しいことを言っていても、またあるときは誤ったことを言っている場合がある。  

- 「あなたは弁護士です」というプロンプト...弁護士にはならないし、正しい法的根拠を持つ回答は返ってくるとは限らない。  
- CLAUDE.mdなどの設定ファイルに決まりごとを書き込む...守られる保証はない。  
- 〇〇を分析して...分析結果が正しい保証はない。ユーザの欲しい答えに寄せてくる。  
- メール返しといて...誤字ってるかもしれないし宛先間違ってるかも。  
- レシート読み込ませて集計...集計結果が正しい保証はない。

精度は上がっていったとしても、LLMである限り動作の正確性は担保できない。  
AIを開発している企業が「弁護士や会計士を滅ぼす」みたいなことを言っているが、原理的に考えて不可能では。   
また、「自然言語で指示してプログラムを書くようになったのは、かつてのコンパイラが登場したときの変化と同じ」と言われるが、  
コンパイラは「どのようなコードをどう変換するか」が厳密に決まっているのに対し、生成AIはいつどのようなコードが書かれるか予測することはできない。
両者には大きな乖離がある。

## ではどう使うべきか
LLMは叩き台作成の高速化には有効。  
ざっくり出してもらったコードを人間が確認し、正確性は従来のソフトウェア（lintなど）で担保するのが良さそう。  
非ITの業務効率化でも同様のことが言える。  
なお、この「従来のソフトウェア」もAIに出してもらえば〜という意見もあると思うが、誤ったものや変更が極めて難しいものが出てくる可能性がある。複雑なものであればあるほどそう  
そういった意味で要件定義・設計・品質保証ができるSWEは必要。
