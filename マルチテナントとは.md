# マルチテナントとは
同じWebサーバーですべての顧客の要求を処理し、同じDBサーバですべての顧客のデータを保存すること。  
反対に、顧客ごとに別々のWebサーバー、DBサーバのインスタンスを割り当てることをシングルテナントという。

![マルチテナントとシングルテナント](/images/マルチテナントとシングルテナント.png)
出典：https://www.onelogin.com/jp-ja/learn/multi-tenancy-vs-single-tenancy

## マルチテナントとシングルテナントの使い分け
マルチテナントの利点は費用・メンテナンスコストの低さ。  
ただし以下の短所がある。
- 一部のテナントがリソースを食い尽くし、ほかのテナントがリソース不足に陥る**ノイジーネイバー問題**  
（これはテナントごとのリソース使用上限を定めることで解決可能）
- 大幅なカスタマイズが難しい
- 他のテナントの情報が漏洩する危険性がある
カスタマイズが必要な場合やセキュリティ要件が厳しい場合はシングルテナント方式が推奨される。  

## マルチテナント方式における情報漏洩対策
- DBで分離する
1. DBをテナントごとに分割する
2. スキーマを分割する
3. 行ごとに分離する
例：PostgresSQLのRow Level Security (RLS)  
2はwhere句でテナントを識別するIDを指定し忘れたりして別テナントの情報が漏洩しないが、代わりにテナントを横断する処理が作りにくい。  
3はスケールしやすく、運用の手間やコストを抑えられるが、where句が抜けると他テナントの情報漏洩につながるため注意が必要。

- アプリケーションで分離する
テナントIDを含まないクエリが実行されると情報漏洩につながるので、そういったコードが書かれないように仕組みで防ぐ。
1. テストコード
2. DB接続をラッパークラスから行うようにし、その使用を強制する
3. ORM・クエリビルダで強制
例：hiberbateの@FilterでtenantId=:tenentIdが含まれているかどうかをチェック
（ただし強制力が弱い、または抜けが発生する可能性があるのでDBで分離を行うのが確実）
